# Dependency Policy & Locking Workflow

**Project:** IRIDIC -- Idiosyncratic Repository of Initialization &
Development Itineraries for Codebases\
**Document:** dependency_policy.md\
**Last Updated:** 2026-02-21

------------------------------------------------------------------------

## 1. Purpose

This document defines:

-   How dependencies are *declared* (source of truth)
-   How dependencies are *selected* (minimal vs optional feature sets)
-   How dependencies are *locked* (reproducible installs via
    `pip-tools`)
-   How dependency changes are *reviewed* and *released*

The policy is optimized for research software where **reproducibility
and clarity beat cleverness**.

------------------------------------------------------------------------

## 2. One Source of Truth (SSOT)

### 2.1 Where dependencies live

**All functional dependency declarations live in `pyproject.toml`**
under:

-   `[project].dependencies` (runtime requirements)
-   `[project.optional-dependencies]` (extras)

This is standardized by PEP 621, and dependency entries must be valid
PEP 508 strings.

### 2.2 What does *not* happen

-   Do **not** maintain a separate hand-written `requirements.txt` as
    the "truth" for a package.
-   Do **not** duplicate runtime dependency intent across multiple
    files.

Instead:

-   `pyproject.toml` expresses **intent**
-   lockfiles express **resolution results** (platform- and
    Python-version--specific)

------------------------------------------------------------------------

## 3. Optional Dependencies ("Extras")

### 3.1 Rationale

Extras are for **non-core** functionality which is real and supported,
but not required for a minimal installation.

Examples: - `web` → Streamlit UI - `dev` → test/lint tooling - `nlp` →
heavy NLP stacks (spaCy, benepar, transformers, models) - `pdf` →
PDF-related tooling

Extras are part of the packaging standard via `Provides-Extra`.

### 3.2 Recommended extras taxonomy

-   `dev`: tests, formatting, linting, type checking
-   `web`: Streamlit + web-only dependencies
-   `docs`: doc build tooling
-   `nlp`: heavy/optional NLP features
-   `all`: convenience meta-extra (optional; use sparingly)

### 3.3 Installation syntax

Extras are requested using square bracket syntax, per PEP 508.
citeturn3search1

Examples:

``` bash
pip install .
pip install ".[dev]"
pip install ".[dev,web]"
```

## 4. pip-tools: Why, and What It Produces

`pip-tools` provides:

-   `pip-compile` → resolves and **pins** all transitive dependencies
-   `pip-sync` → installs an environment to **exactly** match a lockfile

This is ideal for:

-   Streamlit deployments
-   Reproducible local dev environments
-   CI stability
-   "I can reinstall this in six months" guarantees

------------------------------------------------------------------------

## 5. Files and Their Roles

### 5.1 Required files (packaged projects)

-   `pyproject.toml` (SSOT for dependency intent)
-   lockfiles generated by `pip-compile` (pinned resolution)

### 5.2 Optional "input" files

You may use `requirements.in` files **as thin wrappers** for common
install targets.\
These are *not* the SSOT; they are convenience entrypoints for
compilation.

------------------------------------------------------------------------

## 6. Minimal `requirements.in` patterns (copy/paste)

### Pattern A (recommended): compile directly from `pyproject.toml`

This is the clearest and most standard path in `pip-tools`:

``` bash
pip-compile -o requirements.txt pyproject.toml
pip-compile --extra dev -o requirements-dev.txt pyproject.toml
pip-compile --extra web -o requirements-web.txt pyproject.toml
pip-compile --extra dev --extra web -o requirements-dev-web.txt pyproject.toml
```

`pip-tools` explicitly supports compiling both base dependencies and
extras from `pyproject.toml`.

### Pattern B (allowed): a "thin" `requirements.in` that points at the project

This is useful if you strongly prefer a *requirements-first* muscle
memory.

**Minimal** (`requirements.in`):

``` text
.
```

**With extras** (`requirements-dev.in`):

``` text
.[dev]
```

**Multiple extras** (`requirements-dev-web.in`):

``` text
.[dev,web]
```

Then compile:

``` bash
pip-compile -o requirements.txt requirements.in
pip-compile -o requirements-dev.txt requirements-dev.in
pip-compile -o requirements-dev-web.txt requirements-dev-web.in
```

### Pattern C (editable installs)

Use editable installs primarily for **developer environments**:

``` text
-e .
```

or:

``` text
-e .[dev,web]
```

Editable installs remain supported, but pip's underlying editable
mechanism has been evolving; keeping editable usage in dev-only
workflows reduces risk.

**Recommendation:** prefer Pattern A (compile from `pyproject.toml`)
unless you have a strong reason to centralize entrypoints in `.in`
files.

------------------------------------------------------------------------

## 7. Recommended lockfile set

For most repos:

-   `requirements.txt` → runtime only (base)
-   `requirements-dev.txt` → base + `dev`
-   `requirements-web.txt` → base + `web` (if applicable)

Avoid proliferating lockfiles unless you truly have different install
surfaces.

------------------------------------------------------------------------

## 8. Compilation workflow

### 8.1 Setup

``` bash
python -m pip install -U pip pip-tools
```

### 8.2 Compile (base + extras)

(Example using Pattern A)

``` bash
pip-compile -o requirements.txt pyproject.toml
pip-compile --extra dev -o requirements-dev.txt pyproject.toml
pip-compile --extra web -o requirements-web.txt pyproject.toml
```

### 8.3 Sync (developer machine)

``` bash
pip-sync requirements-dev.txt
```

## 9. Platform specificity warning

`pip-tools` lockfiles can vary by OS / Python version because
conditional dependencies resolve differently (e.g., Windows-only deps).

Policy options:

-   **Single-platform projects:** one lockfile set is fine.
-   **Multi-platform support:** maintain per-platform lockfiles *or*
    switch to a resolver that supports universal resolution (e.g.,
    `uv`).

IRIDIC default: **lock on the platform you deploy on** (often Linux for
Streamlit) and document it.

## 10. How strict should version specs be in `pyproject.toml`?

### 10.1 Default stance (research tooling)

-   Use lower bounds (`>=`) where needed.
-   Use upper bounds (`<`) where breakage is plausible.
-   Avoid hard pins (`==`) in `pyproject.toml` unless there is a known
    incompatibility.

Rationale: - Pins belong in lockfiles. - `pyproject.toml` should
describe compatibility intent, not freeze the universe.

### 10.2 Heavy stacks belong behind extras

For ecosystems like CLATR/ALASTR/RASCAL:

-   Keep heavyweight NLP dependencies out of the base install.
-   Put them in optional extras (e.g., `nlp`). citeturn0search3

This keeps installs lighter, reduces conflicts, and prevents forcing
users into heavyweight dependency trees unnecessarily.

## 11. Commit discipline for dependency changes

Any dependency changes must include:

-   The `pyproject.toml` change (intent)
-   Regenerated lockfiles (resolution)
-   A commit message which states *why* (bugfix, compatibility, new
    feature)

Do not "sneak" dependency updates inside unrelated commits.
